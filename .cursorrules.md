# LawMarket ‚Äî Cursor Rules

> Rename this file to `.cursorrules` and place it at the project root.
> Full architecture context lives in `LawMarket-Master-Dev-Spec.md` ‚Äî read it if it's in the repo.

## Project Context

LawMarket is a premium legal-services marketplace for Georgia. Law providers create a profile, list services, and ‚Äî once manually verified by an admin ‚Äî list successful cases. Clients browse, message, and book, including a separate "expat consultation" program for foreigners setting up a business. Stack: Next.js (App Router, TypeScript), Supabase (Postgres/Auth/Storage/Realtime), Tailwind CSS, shadcn/ui.

---

## Coding Standards

- TypeScript strict mode everywhere. No `any`. No `@ts-ignore` without a comment explaining why it's actually safe.
- Functional components only ‚Äî no class components.
- Keep components under ~150 lines. When one grows past that, extract a subcomponent or a hook rather than letting it sprawl.
- Default to Server Components. Add `'use client'` only when you need interactivity, browser APIs, or hooks ‚Äî and put it on the smallest leaf component that needs it, not the whole page.
- All Supabase access goes through `lib/data/*.ts`. Never call `supabase.from()` directly inside a component or page.
- Mutations go through Server Actions where practical, validated with Zod before they touch the database.
- Naming: `PascalCase` components, `camelCase` functions/variables, `kebab-case` file and folder names, `SCREAMING_SNAKE_CASE` for true constants.
- One component per file. Small subcomponents can live in the same folder, not the same file.
- No `console.log` in committed code.
- Every exported function has an explicit return type ‚Äî don't rely on inference for anything public.

## Styling Rules ‚Äî "Premium Minimalism"

- Tailwind utility classes only. No `.css`/`.module.css` files except `globals.css` for `@font-face` and CSS variable definitions.
- **4px grid:** use Tailwind's default spacing scale exclusively. Never an arbitrary value like `mt-[13px]`.
- **Borders over shadows:** a 1px hairline border (`border border-border`) is the default way to separate cards and sections. `box-shadow` is reserved for genuinely elevated surfaces ‚Äî modals, popovers, dropdowns ‚Äî never a plain content card.
- **Radius:** small and consistent (`rounded-sm` / `rounded-md`, ~2‚Äì6px). Sharp enough to feel editorial, not a rounded "app" look. `rounded-full` is reserved for avatars and the verification seal.
- **Numbered markers** (01 / 02 / 03) only for genuinely sequential content ‚Äî a real step-by-step flow ‚Äî never as decoration on a features grid.
- The **verification seal** (`components/marketplace/VerifiedSeal.tsx`) is the one recurring signature motif. Use it consistently for verified providers and verified cases. Don't introduce a second, competing "trust" visual language (e.g. a generic checkmark badge) elsewhere in the product.
- **Layout:** long-form content (bios, case summaries) capped near 72 characters per line for readability. Page shell `max-w-6xl` to `max-w-7xl`. Avoid a permanently centered, symmetric hero ‚Äî prefer the headline in tension with a supporting element (a stat, the seal).

## Security Rules

- **RLS is the real security boundary**, not application code. Every table has RLS enabled, default-deny. Nothing is queryable until a policy explicitly allows it.
- `lib/data/*.ts` exists for consistency and auditability, not as the security check itself. Its real job: guarantee every user-facing query runs on the authenticated client (anon key + user JWT) so RLS actually applies, and keep raw Supabase calls out of components.
- The `service_role` key **bypasses RLS entirely**. It is only ever used in isolated, trusted server contexts ‚Äî admin verification actions, payment webhook handlers ‚Äî via `lib/supabase/admin.ts`, and is never imported into any file reachable from a Client Component or the browser bundle.
- **Never expose a raw database UUID in a public-facing URL.** Provider profiles use `profiles.public_slug` (`/providers/nino-beridze-4f2a`), not `/providers/[uuid]`. There is no public route for client profiles at all.
- Uploaded documents are private by default. Case evidence and, especially, signed client-consent documents live in private Storage buckets, served only via short-lived signed URLs to the uploading provider and admin reviewers. The consent document itself never becomes public ‚Äî only the verified case *summary* does.
- Validate on the server, always. Client-side Zod validation is for UX; server-side validation in the Server Action or route handler is the actual gate. Never trust a value because the client already checked it.
- Payment webhook handlers verify the gateway's signature before writing anything, and are idempotent ‚Äî a retried webhook must never double-charge a subscription or double-confirm a booking.
- Admin routes are gated twice: RLS on every query they make, *and* a role check in the route/layout itself. Don't rely on RLS alone to keep a non-admin from even seeing the admin shell render.

## Do / Don't Quick Reference

| Do | Don't |
|---|---|
| Query via `lib/data/*.ts` | Call `supabase.from()` inside a component |
| Use `profiles.public_slug` in URLs | Put a raw `uuid` in a public route |
| Keep `service_role` in `lib/supabase/admin.ts` only | Import the service-role client anywhere client-reachable |
| Add a new offering as a `programs` row | Hardcode a new service type as an enum value |
| Hairline border on cards | Drop-shadow on cards |
| `Noto Serif Georgian` fallback on headlines | Ship a headline font with no Georgian glyphs and hope |
